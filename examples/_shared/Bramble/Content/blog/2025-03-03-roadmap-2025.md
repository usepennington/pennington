Roadmaps are promises with error bars. I'd rather be specific about what we're aiming at and transparent about the uncertainty than give you a vague list of themes that could mean anything. So here's what we're actually planning for 2025, in rough priority order, with honest caveats about each.

## Bramble 2.0

The headline for 2025 is a 2.0 language revision. The version number signals intent: there will be a small number of breaking changes, all of them deliberate, and we will provide migration tooling (`sprig migrate`) so the changes are mechanical.

The changes we're committed to:

- **Trait coherence rules** tightened. The current orphan rules are too permissive and cause surprising behavior when two packages both try to implement a trait for a foreign type. We're aligning with a stricter model.
- **`Task<T>` as a language construct.** Right now `Task` is a standard-library type. In 2.0, `spawn`/`await` will be language keywords with first-class scheduler integration rather than functions over a library type. The user-visible difference is small; the implementation difference is significant and enables better diagnostics.
- **`?` propagation across `await`.** The missing piece from 1.2. `fetch(url).await?` should work. It will in 2.0.

What we're *not* changing: the module system, the ownership model, and the syntax for pattern matching. Those are stable.

## Faster Thicket resolver

Dependency resolution in Thicket is correct but slow on large lockfiles. The current resolver is a backtracking SAT approach that works well for small dependency trees and degrades badly when packages have deep transitive graphs. We're replacing the resolution algorithm with a version-union approach that avoids most backtracking by construction. Internal benchmarks on large synthetic trees show 8–12x speedup; real projects should see 3–5x on a clean resolve.

You can read about how Thicket currently handles versioning in the [versioning explanation](xref:bramble.explanation.versioning-in-thicket). The new resolver won't change the semantics — only the speed.

## Diagnostics overhaul

Bramble's error messages have always been *correct* but not always *useful*. The type-mismatch message, in particular, has a reputation for pointing at the symptom rather than the cause. We're investing in a dedicated diagnostics pass that has access to the full inference graph and can trace a mismatch back to where the conflicting assumption was first introduced.

The goal is errors that read like explanations, not accusations.

```text
error[E0031]: type mismatch
  --> src/main.brm:14:5
   |
 8 |     let scores: List<Int> = load_scores(path)?
   |                 --------- expected element type inferred here
14 |     scores.push("unranked")
   |     ^^^^^^^^^^^^^^^^^^^^^^^ found `Str`, expected `Int`
   |
   = note: the element type was fixed at line 8 when `scores` was annotated
```

That's the target quality. We're not there uniformly yet.

## Timeline

2.0 preview in Q4 2025, stable in early 2026. Thicket resolver lands as a Thicket 2.0 release, tracked separately from the language version. Diagnostics improvements ship incrementally across minor releases throughout the year — you'll see them in 1.3 and 1.4 before 2.0 arrives.

As always, nothing here is a guarantee. If something turns out to be harder than expected, I'll write about it. That's more useful than pretending the plan was always the plan.
