Thicket 2.0 is out, and the headline is a dependency resolver that's substantially faster for large projects. But there are also a couple of behavioral changes worth knowing about before you upgrade, so read this before you run `thicket self-update`.

## Why the resolver needed replacing

Thicket 1.x used a backtracking SAT-based resolver. This is a well-understood approach that gives correct results, but its worst-case behavior is exponential in the number of conflicting constraints. For small projects with shallow dependency trees — which was everyone, in 2023 — it was fast enough that nobody noticed. As the Thicket registry grew and packages accumulated transitive dependencies, some `thicket install` runs on clean CI caches started taking tens of seconds. A few outlier projects in the community reported over a minute.

The new resolver uses a version-union algorithm. Rather than backtracking on conflict, it builds the full set of compatible version ranges for each package in one forward pass and finds the intersection. Backtracking is only needed when the intersection is empty — a genuine version conflict — at which point the resolver produces a diagnostic explaining the conflict rather than a timeout.

Benchmarks on the corpus of real `bramble.toml` files from Thicket's public registry:

```text
Project size       | v1.x median | v2.0 median | Speedup
Small  (<20 deps)  |    0.4 s    |    0.3 s    |   1.3x
Medium (20–80)     |    2.1 s    |    0.5 s    |   4.2x
Large  (80+)       |   18.4 s    |    1.9 s    |   9.7x
```

The speedup is most dramatic at the high end, which is exactly where it matters most.

## Stricter lockfile semantics

Thicket 2.0 treats `thicket.lock` as the authoritative source for installed versions. In 1.x, `thicket install` would silently upgrade a locked dependency if the resolver found a newer compatible version and the lockfile was "old enough." That behavior surprised people and made reproducible builds unreliable.

In 2.0: if a `thicket.lock` exists, `thicket install` installs exactly what's in it. To update dependencies, you run `thicket update` (all) or `thicket update <package>` (one). To regenerate the lockfile from scratch, you delete it and run `thicket install`.

This matches how most developers expected it to work anyway. The [pinning and updating how-to](xref:bramble.how-to.thicket.pin-and-update-versions) has been updated to reflect the new model.

## Migration

For most projects, upgrading is:

```bash
thicket self-update
thicket install
```

The new resolver may produce a different set of selected versions than the old one on an existing lockfile-free project, because it uses a different preference ordering when multiple valid solutions exist. If you have a lockfile already committed, the installed versions won't change — the lockfile is honored exactly.

Two edge cases to watch for:

- **Circular dev-dependencies.** Thicket 2.0 now rejects circular dependencies in `[dev-dependencies]` sections. Thicket 1.x silently dropped one edge. If you see a new `E_CIRCULAR_DEP` error, that's a real problem in your config that was previously hidden.
- **Custom registry entries.** If you use `[[source]]` entries pointing at private registries, re-authenticate after the update. The credential format changed to support scoped tokens.
