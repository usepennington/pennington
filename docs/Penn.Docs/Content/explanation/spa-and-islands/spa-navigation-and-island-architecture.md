Explain the SPA-without-Blazor-WASM architecture. Penn doesn't ship a client-side framework — instead, it uses server-rendered HTML fragments (islands) swapped into the DOM by a small client-side script. Discuss why: Blazor WASM adds significant download size and complexity for what is fundamentally a content site. Walk through the `SpaEnvelope` data contract — JSON with page title, description, outline, and a dictionary of island name → rendered HTML. Explain the client-side lifecycle: intercept link clicks, fetch `/_spa-data/{slug}.json`, match islands by `data-spa-island` attribute, swap innerHTML, update document title and meta tags, push browser history. Cover loading strategies: `keep` (don't touch the island during loading), `skeleton` (show a placeholder after 100ms), `clear` (empty immediately). Explain the timing contract — why the 100ms skeleton delay and 250ms minimum skeleton display exist (to prevent flash for fast loads). Discuss view transitions API integration and accessibility (ARIA live regions, focus management). Explain why xrefs must be resolved in the SPA data endpoint separately from the response processor chain — the SPA endpoint returns raw HTML fragments, not full pages processed by middleware.
